# CacheClient 上线改造清单

## 目标

将 `src/main/java/com/hmdp/utils/CacheClient.java` 从教学/实验实现提升为可上线版本，重点覆盖：

- 缓存穿透
- 热点 key 击穿
- 分布式锁安全性
- 异步重建稳定性
- 可观测性
- 故障降级

## 当前结论

当前实现不满足上线要求。主要问题：

- 异步重建的 double check 逻辑错误
- 分布式锁释放不安全
- 线程池使用无界队列
- 异步任务没有日志和指标
- 逻辑过期方案依赖预热，边界没有明确约束

## P0 必改项

### 1. 修复逻辑过期的 double check

问题：

- 当前代码中的二次检查无效，无法判断缓存是否已被其他线程或其他实例刷新。

要求：

- 异步任务启动后重新读取 Redis
- 重新解析 `RedisData`
- 如果缓存已经是新鲜数据，则直接退出，不再查库

验收标准：

- 同一个 key 在并发重建场景下，最多只有一次有效 DB 回源
- 不会因为重复重建把新值覆盖成旧值

### 2. 重构分布式锁为安全解锁方案

问题：

- 当前锁值固定为 `"1"`
- 解锁直接 `delete(key)`，会误删其他线程/实例的新锁

要求：

- 加锁时写入唯一 token
- 解锁时用 Lua 脚本做 compare-and-delete
- token 建议包含实例标识和线程标识

验收标准：

- 锁过期后被其他线程重新获取时，旧线程不会误删新锁
- 并发压测下不会出现重复重建失控

### 3. 改造异步重建线程池

问题：

- 当前线程池使用无界 `LinkedBlockingQueue`
- 热点集中时会造成任务无限堆积

要求：

- 使用有界队列
- 配置明确的线程名
- 设置拒绝策略
- 线程池参数从配置文件读取或至少集中常量化

建议结构：

- `corePoolSize`
- `maximumPoolSize`
- `queueCapacity`
- `keepAliveTime`
- `threadNamePrefix`
- `rejectionHandler`

验收标准：

- 高峰流量下不会因为任务堆积导致内存持续增长
- 队列满时系统仍然可以返回旧值，不阻塞主请求

### 4. 完善异步任务异常治理

问题：

- 当前异步任务中 `throw new RuntimeException(e)` 没有线上价值
- 缺少日志、指标、失败上下文

要求：

- 捕获异常后记录结构化日志
- 日志至少包含 `key`、`id`、异常摘要、耗时
- 无论成功失败都必须释放锁

验收标准：

- 重建失败时请求仍可返回旧值
- 日志中能够定位失败 key 和失败原因

## P1 应改项

### 5. 增加缓存层指标与监控

要求：

- 命中次数
- 未命中次数
- 逻辑过期次数
- 重建触发次数
- 重建成功次数
- 重建失败次数
- 获取锁成功/失败次数
- 线程池队列长度

验收标准：

- 可以基于指标判断缓存是否健康
- 可以识别是否存在重建堆积或锁竞争异常

### 6. 明确 `queryWithLogicalExpire` 的使用边界

问题：

- 当前实现依赖缓存预热
- 如果调用方误以为它会自动回源，会造成业务误判

要求：

- 在方法注释中明确：该方案仅适用于已预热的热点数据
- 调用方必须保证 key 已存在，或在外层增加首次加载逻辑

验收标准：

- 方法文档清楚说明适用场景
- 调用方不会误用该方法处理冷数据

### 7. 统一空值策略

问题：

- 单对象查询和列表查询都支持穿透保护
- 但 `List` 场景下空列表是否当作空值，需要统一约定

要求：

- 明确 `null` 与空列表的业务语义
- 统一是否缓存空列表

验收标准：

- 同一类查询不存在一处缓存空列表、一处不缓存的行为分裂

### 8. 统一 key 与锁 key 规范

要求：

- 统一业务 key 前缀命名
- 统一锁 key 前缀命名
- 避免把 `LOCK_SHOP_KEY` 这种业务耦合常量直接复用到通用组件

验收标准：

- `CacheClient` 不直接依赖具体业务类型命名
- 调用方可以显式传入 lock key 前缀或由通用策略生成

## P2 优化项

### 9. 增加随机 TTL 抖动

适用：

- `queryWithPassThrough`
- `queryWithPassThroughList`

目的：

- 降低大量 key 同时过期带来的雪崩风险

验收标准：

- 缓存过期时间分布不再集中于固定时刻

### 10. 统一日志语义

建议日志节点：

- 缓存命中
- 缓存 miss
- 逻辑过期
- 获取锁成功
- 获取锁失败
- 异步重建成功
- 异步重建失败

要求：

- 高频成功日志使用 `debug` 或采样
- 失败日志使用 `warn`/`error`

### 11. 组件命名修正

问题：

- 历史类名存在拼写错误

要求：

- 更名为 `CacheClient`
- 统一调用方引用

说明：

- 这是可维护性问题，不是功能性 blocker

## 推荐实施顺序

1. 修复 double check
2. 重构分布式锁
3. 改造线程池
4. 完善异常处理与日志
5. 加指标监控
6. 明确方法边界与空值策略
7. 处理 TTL 抖动和命名问题

## 上线前验收用例

### 并发场景

- 单热点 key 过期，1000 并发请求进入
- 预期只有一个后台任务回源 DB
- 其他请求返回旧值，不阻塞

### 锁安全场景

- 模拟锁超时后其他线程重新获取锁
- 预期旧线程不会删掉新线程的锁

### 线程池压力场景

- 模拟大量热点 key 同时过期
- 预期线程池队列受控，无内存持续增长

### 异常场景

- 模拟 DB 查询异常
- 预期旧值仍可返回，锁正常释放，错误日志完整

### 冷数据场景

- 对未预热 key 调用 `queryWithLogicalExpire`
- 预期行为与文档一致，不出现误判

## 交付标准

满足以下条件后再考虑上线：

- 并发压测通过
- 锁安全性验证通过
- 线程池和 Redis/DB 指标可观测
- 日志可定位重建失败原因
- 方法边界和调用约束已经文档化
