## 1. 文档目标

本需求文档用于指导研发完成“博客点赞/取消点赞”功能的业务增强。

目标不是重做现有点赞功能，而是在**保留现有点赞主链路可用**的基础上，补齐以下能力：

- 点赞/取消点赞事件标准化
- Kafka 事件发送
- 异步消费者处理
- 作者通知能力预留
- 热度统计能力预留
- 消费幂等能力

本文档默认你是实现该功能的程序员，重点给出：

- 业务目标
- 范围说明
- 交互与行为规则
- 事件定义
- 接口及服务侧实现要求
- 验收标准

## 2. 业务背景

当前项目已经支持博客点赞与取消点赞，逻辑主要在：

- `src/main/java/com/hmdp/service/impl/BlogServiceImpl.java`

当前实现已完成：

- 数据库点赞数字段增减
- Redis 点赞状态维护

但当前实现存在扩展瓶颈：

1. 点赞成功后若继续增加通知、热度、画像、统计等逻辑，会让主链路越来越重
2. 主流程与后置动作耦合过深，不利于后续扩展
3. 后续若增加多个下游功能，会不断修改核心点赞方法

因此，需要引入 `blog.liked` 事件，把点赞后的后置动作从主链路中剥离。

## 3. 产品目标

### 3.1 总目标

将“点赞/取消点赞”抽象成一个业务事件，在主链路成功后投递 Kafka，由多个下游异步消费。

### 3.2 本期目标

本期必须完成：

- 点赞成功后发送 `blog.liked` 事件
- 取消点赞成功后发送 `blog.liked` 事件
- 事件中区分 `LIKE` 和 `UNLIKE`
- 消费者至少实现一类实际消费逻辑
- 消费者具备基础幂等能力

### 3.3 本期不要求完成

本期不强制要求：

- 完整的站内信系统
- 完整推荐系统
- 死信队列完整治理
- 后台配置页或管理台页面
- 事件回放平台

## 7. 核心业务流程

### 7.1 点赞流程

1. 用户点击点赞
2. 服务端校验当前用户是否已点赞
3. 若未点赞：
   - 数据库点赞数 `+1`
   - Redis 点赞 ZSet 新增当前用户
4. 点赞主链路成功后，发送 `blog.liked` 事件，`action=LIKE`
5. 接口立即返回成功
6. Kafka 下游消费者异步处理后续逻辑

### 7.2 取消点赞流程

1. 用户点击取消点赞
2. 服务端校验当前用户是否已点赞
3. 若已点赞：
   - 数据库点赞数 `-1`
   - Redis 点赞 ZSet 移除当前用户
4. 主链路成功后，发送 `blog.liked` 事件，`action=UNLIKE`
5. 接口立即返回成功
6. Kafka 下游消费者异步处理

---

## 8. 业务规则

### 8.1 点赞状态判定

系统以 Redis 的点赞 ZSet 为准判断当前用户是否已点赞。

### 8.2 幂等要求

同一条 Kafka 事件被消费多次，不允许造成重复业务副作用。

### 8.3 事件发送时机

只有当点赞主链路成功后，才允许发送 Kafka 事件。

### 8.4 用户未登录

未登录用户不能点赞，也不能触发点赞事件。

### 8.5 不允许重复点赞

用户对同一篇博客重复点赞时，不得重复增加点赞数。

### 8.6 取消点赞前提

只有已点赞状态下才能取消点赞。

## 9. 事件定义

### 9.1 Topic

```text
blog.liked
```

### 9.2 事件对象

推荐字段如下：

| 字段 | 类型 | 是否必须 | 说明 |
|---|---|---:|---|
| eventId | String | 是 | 事件唯一标识，用于幂等 |
| blogId | Long | 是 | 博客 id |
| authorId | Long | 是 | 博客作者 id |
| userId | Long | 是 | 执行点赞/取消点赞的用户 id |
| action | String | 是 | `LIKE` / `UNLIKE` |
| occurredAt | Long | 是 | 事件发生时间戳 |

### 9.3 分区 key

建议使用：

```text
blogId
```

原因：

- 同一博客的点赞事件尽量路由到同一分区
- 有利于局部顺序消费

## 10. 消费者规划

### 10.1 本期必须实现的消费者

至少落地一个可运行的消费者，建议优先做：

#### `BlogLikeNotifyConsumer`

职责：

- 消费 `blog.liked`
- 对 `LIKE` 动作执行通知逻辑
- 当前阶段可先打印日志或写轻量记录

说明：

- `UNLIKE` 动作本期可忽略通知
- 后续可以扩展为站内信

### 10.2 本期预留但可暂缓实现的消费者

#### `BlogHotScoreConsumer`

职责：

- 按点赞/取消点赞更新博客热度模型

#### `UserBehaviorConsumer`

职责：

- 记录点赞行为流水
- 为画像与推荐做准备

## 11. 接口与后端行为要求

### 11.1 Controller 层

现有点赞接口继续沿用，不新增前端接口。

### 11.2 Service 层要求

在点赞成功和取消点赞成功后：

- 构造 `BlogLikedEvent`
- 发送 Kafka 消息

### 11.3 查询作者信息

发送事件前应能拿到博客作者 id。

若当前点赞方法里没有作者信息，研发需要补上最小查询逻辑。

### 11.4 失败处理

主业务优先级高于消息后处理。

本期允许采用如下策略：

- 主链路成功
- Kafka 发送失败时记录 error 日志

说明：

本期不强制实现 outbox，但代码结构应保留后续升级空间。

---

## 12. 幂等要求

### 12.1 幂等目标

同一事件重复消费时，不允许：

- 重复通知
- 重复加热度
- 重复写行为流水

### 12.2 本期建议方案

建议基于：

- `eventId`
- Redis `SETNX`

实现消费者幂等。

推荐 key 形式：

```text
mq:idempotent:blog:liked:{consumerName}:{eventId}
```

### 12.3 过期时间

建议设置合理过期时间，例如：

- 24 小时
- 48 小时

由研发按当前项目实际情况确定。

## 14. 验收标准

### 14.1 点赞主流程

当用户点赞博客时：

- 数据库点赞数增加
- Redis 点赞状态存在
- 接口返回成功

### 14.2 取消点赞主流程

当用户取消点赞时：

- 数据库点赞数减少
- Redis 点赞状态移除
- 接口返回成功

### 14.3 Kafka 事件发送

点赞成功后能发送 `blog.liked`，且事件内容完整。

取消点赞成功后也能发送 `blog.liked`，且 `action=UNLIKE`。

### 14.4 消费者可运行

消费者能正常收到消息并执行逻辑。

### 14.5 幂等有效

同一事件重复投递时，不会产生重复副作用。

### 14.6 非登录用户

未登录用户点赞时，不允许发送事件。

## 18. 结论

本期实现的关键不是“再加一个 Kafka topic”，而是把点赞主链路和后置动作拆开，建立事件化能力。

这会让后续：

- 通知
- 热度
- 推荐
- 画像
- 统计

都可以在不侵入主点赞逻辑的前提下扩展。

一句话总结：

> 点赞成功本身仍然同步完成，而点赞之后的附加动作由 `blog.liked` 事件驱动异步处理。
